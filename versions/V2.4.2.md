# ZNAMBO Telegram Assistant V2.4.2

Release date: 2026-06-09.

## Scope

V2.4.2 fixes transactional reminder semantics and scheduler deduplication without replacing the
V2 Smart AI Planner.

## Changes

- Added a distributed 55-second `reminder_runner` lease in `assistant.runtime_locks`.
- Enforced runner order: lease, reconcile, claim, send, record, advance, refresh, release.
- Anchored interval schedules to `policy.startsAt`, never to actual delivery time.
- Made snooze a separate one-off reminder and resumed the next regular anchor-grid slot.
- Added strict end-window expiration and `onWindowEnd`, defaulting to `expire_silently`.
- Prevented interval tasks from inheriting post-event menus.
- Added atomic ActionPlan commit for items, policies, initial reminders and occurrences.
- Added pre-commit policy validation and transaction telemetry.
- Normalized the Central Park request into exactly two events and four daily reminder policies.
- Kept vague day-part validation while accepting explicit ranges such as `—É—ā—Ä–ĺ–ľ —Ā 8 –ī–ĺ 12`.
- Normalized open-ended Drik requests into one task and one bounded `nag_until_ack` policy.
- Classified tomorrow reminders under `–°–ļ–ĺ—Ä–ĺ`, keeping recurring/long-term policies under `–Ē–į–Ľ—Ć–Ĺ–ł–Ķ`.
- Added owner-only `/admin_repair_v242 preview|apply`.
- Added a protected self-cleaning production probe for snooze/grid acceptance.
- Probe cleanup explicitly removes temporary reminders, policies and items.
- Added a protected read-only dashboard snapshot for production acceptance.
- Repair targets orphan Drik tasks, malformed interval-as-recurring Drik policies and the exact
  partial Central Park records from the failed production transaction.
- Added `/versiondebug` and safe scheduler/version fields to `/api/health`.

## Database

Migration: `drizzle/0005_transactional_reminder_semantics.sql`.

It adds:

- `assistant.reminder_policies.on_window_end`;
- `assistant.runtime_locks`;
- a unique occurrence-to-reminder index;
- a runtime lock expiry index.

The migration is idempotent and was applied to production before code deployment.
